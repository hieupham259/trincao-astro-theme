Modern full-stack applications do not have to span multiple services, infrastructure providers, or deployment pipelines. This tutorial walks through building a complete, authenticated todo application — React frontend, REST API, relational database, session store, and AI features — deployed as a single Cloudflare Worker. Every piece of the stack runs at the edge.

## What We Are Building

The application is a personal productivity tool with three core sections: a daily **Todo** view (three-day rolling calendar), a **Backlog** for tasks without a fixed date, and a **Stats** dashboard that computes a weighted effectiveness score. Users authenticate with email/password or Google OAuth. Workers AI provides two AI features: automatic todo categorization and task breakdown into subtasks.

The full tech stack:

- **Frontend**: React 19, React Router v7, TanStack Query v5, Tailwind CSS v4
- **Backend**: Hono (TypeScript), running as a Cloudflare Worker
- **Database**: Cloudflare D1 (edge SQLite), schema managed with Drizzle ORM
- **Auth**: better-auth with a custom PBKDF2 password hasher
- **Session store**: Cloudflare Workers KV (also used for rate limiting)
- **AI**: Cloudflare Workers AI — Llama 3.3 70B Instruct
- **Build tool**: Vite + `@tailwindcss/vite`
- **Testing**: Vitest + `@cloudflare/vitest-pool-workers`, MSW, Testing Library

## Project Structure

The project is split into two roots that share one deployment unit:

```
src/           # Cloudflare Worker (API + static asset binding)
  index.ts     # Hono app entry — middleware, route mounting, error handling
  types.ts     # Bindings and AppVariables types
  db/
    schema.ts  # Drizzle table definitions
    client.ts  # getDb() helper
  routes/      # todos.ts | backlog.ts | stats.ts | ai.ts | auth.ts
  middleware/  # auth.ts (session + requireAuth) | logger.ts | rate-limit.ts
  lib/         # better-auth.ts | date.ts | errors.ts

web/           # Vite SPA (React)
  src/
    pages/     # TodosPage | BacklogPage | StatsPage | LoginPage
    components/
    lib/       # api.ts (Hono RPC client) | date.ts

migrations/    # Drizzle-generated SQL applied by wrangler d1 migrations apply
```

At build time, `vite build` outputs static assets into `./public`. The Worker's `assets` binding in `wrangler.jsonc` then serves those files directly, and `not_found_handling: "single-page-application"` handles client-side routing fallback — no separate CDN or static hosting required.

## Key Concepts

### Bindings: One Config, All Services

All Cloudflare services are declared as bindings in `wrangler.jsonc`. The `types.ts` file mirrors them into a `Bindings` type consumed by Hono's generic:

```typescript
// src/types.ts
export type Bindings = {
  AI: Ai;
  DB: D1Database;
  SESSIONS: KVNamespace;
  ASSETS: Fetcher;
  BETTER_AUTH_SECRET: string;
  BETTER_AUTH_URL: string;
  GOOGLE_CLIENT_ID?: string;
  GOOGLE_CLIENT_SECRET?: string;
};

export type AppVariables = {
  user: { id: string; email: string; name: string | null } | null;
  session: { id: string; expiresAt: Date } | null;
  requestId: string;
};

export type AppEnv = { Bindings: Bindings; Variables: AppVariables };
```

Every route handler receives `c.env.DB`, `c.env.AI`, and `c.env.SESSIONS` as fully-typed objects — no SDK instantiation, no connection strings.

### Schema and Migrations with Drizzle ORM

The database schema is defined once in TypeScript and Drizzle Kit generates the SQL migration files. The key tables are `todo`, `backlog_item`, `tag`, and `todo_tag`, plus the four auth tables generated by better-auth (`user`, `session`, `account`, `verification`).

Notice the `todo` table uses a plain text `scheduled_date` column (format `YYYY-MM-DD`) rather than a timestamp — this avoids timezone-shift bugs when grouping by calendar day across different user locales:

```typescript
// src/db/schema.ts (abbreviated)
export const todo = sqliteTable(
  'todo',
  {
    id: text('id').primaryKey(),
    userId: text('user_id').notNull().references(() => user.id, { onDelete: 'cascade' }),
    title: text('title').notNull(),
    description: text('description'),
    scheduledDate: text('scheduled_date').notNull(),  // YYYY-MM-DD, timezone-safe
    priority: text('priority', { enum: ['high', 'med', 'low'] }).notNull().default('med'),
    status: text('status', { enum: ['open', 'done', 'archived'] }).notNull().default('open'),
    completedAt: integer('completed_at', { mode: 'timestamp_ms' }),
    aiCategory: text('ai_category'),
    createdAt: integer('created_at', { mode: 'timestamp_ms' }).notNull(),
    updatedAt: integer('updated_at', { mode: 'timestamp_ms' }).notNull(),
  },
  (t) => ({
    userDateIdx: index('idx_todo_user_date').on(t.userId, t.scheduledDate),
    userStatusIdx: index('idx_todo_user_status').on(t.userId, t.status, t.updatedAt),
  }),
);
```

### Route Design and Zod Validation

Each feature area lives in its own Hono sub-app mounted via `api.route('/todos', todosRoutes)`. Routes validate input with `@hono/zod-validator` before touching the database. The `listSchema` for `GET /api/todos` enforces a maximum 31-day date range at the validation layer so the database never receives unbounded queries:

```typescript
// src/routes/todos.ts (list + create handlers)
const listSchema = z
  .object({ from: dateStr, to: dateStr })
  .refine((v) => v.from <= v.to, 'from must be <= to')
  .refine((v) => {
    const days = (Date.parse(`${v.to}T00:00:00`) - Date.parse(`${v.from}T00:00:00`)) / 86400000;
    return days <= 31;
  }, 'range max 31 days');

todosRoutes.get('/', zValidator('query', listSchema), async (c) => {
  const db = getDb(c.env.DB);
  const u = c.get('user')!;
  const { from, to } = c.req.valid('query');
  const rows = await db
    .select()
    .from(todo)
    .where(and(eq(todo.userId, u.id), between(todo.scheduledDate, from, to)))
    .orderBy(desc(todo.priority), desc(todo.updatedAt));
  return c.json({ todos: rows });
});
```

### Authentication with better-auth and PBKDF2

better-auth handles session management and OAuth flows, but its default scrypt password hasher exceeds the 10 ms CPU budget on the Cloudflare Workers free plan. The solution is to swap in a PBKDF2-SHA256 hasher built on the Web Crypto API, which runs in ~3–5 ms and still meets OWASP's minimum iteration recommendations. The `better-auth.ts` module caches the auth instance per D1 binding reference to avoid repeated initialization within the same isolate.

Workers KV serves double duty: better-auth stores session tokens there for fast lookup, and a custom rate-limit middleware uses it as a sliding counter keyed by `cf-connecting-ip`:

```typescript
// src/middleware/rate-limit.ts
export const rateLimit = (opts: { key: string; limit: number; windowSec: number }) =>
  createMiddleware<AppEnv>(async (c, next) => {
    const ip = c.req.header('cf-connecting-ip') ?? 'anon';
    const key = `rl:${opts.key}:${ip}`;
    const cur = Number((await c.env.SESSIONS.get(key)) ?? 0);
    if (cur >= opts.limit) return c.json({ error: 'Too many requests' }, 429);
    await c.env.SESSIONS.put(key, String(cur + 1), { expirationTtl: opts.windowSec });
    await next();
  });
```

### Workers AI: Categorization and Streaming Breakdown

The `/api/ai` routes call Cloudflare's AI binding directly — no third-party API key, no additional billing account. `POST /api/ai/categorize` returns structured JSON (category, priority, tags) for pre-filling the todo form. `POST /api/ai/breakdown` streams a bullet-list of subtasks back to the browser as a Server-Sent Events response:

```typescript
// src/routes/ai.ts
aiRoutes.post('/categorize', zValidator('json', categorizeSchema), async (c) => {
  const { title, description } = c.req.valid('json');
  const res = await c.env.AI.run('@cf/meta/llama-3.3-70b-instruct', {
    messages: [
      {
        role: 'system',
        content:
          'Classify a todo. Respond JSON only: {"category": string, "priority": "high"|"med"|"low", "tags": string[]}',
      },
      { role: 'user', content: `Title: ${title}\nDesc: ${description ?? ''}` },
    ],
    response_format: { type: 'json_object' },
  });
  return c.json(res);
});

aiRoutes.post('/breakdown', zValidator('json', categorizeSchema), async (c) => {
  const { title } = c.req.valid('json');
  const stream = await c.env.AI.run('@cf/meta/llama-3.3-70b-instruct', {
    messages: [
      { role: 'system', content: 'Break the task into 3-6 actionable subtasks as bullet list.' },
      { role: 'user', content: title },
    ],
    stream: true,
  });
  return new Response(stream as unknown as ReadableStream, {
    headers: { 'content-type': 'text/event-stream' },
  });
});
```

## Getting Started

You need Node.js 18+, a Cloudflare account with Workers and D1 enabled, and npm.

```bash
# 1. Install dependencies
npm install

# 2. Authenticate Wrangler
npx wrangler login

# 3. Create the D1 database and paste the returned database_id into wrangler.jsonc
npx wrangler d1 create your-todo-db

# 4. Create the KV namespace for sessions and paste the returned id into wrangler.jsonc
npx wrangler kv namespace create SESSIONS

# 5. Generate Cloudflare binding types
npm run cf-typegen

# 6. Apply schema migrations to the local D1 database
npm run db:apply:local
# (Runs: wrangler d1 migrations apply your-todo-db --local)

# 7. Create a .dev.vars file for local secrets (never commit this file)
# BETTER_AUTH_SECRET=your-random-32-byte-secret-here
# BETTER_AUTH_URL=http://localhost:8787
# GOOGLE_CLIENT_ID=your-google-client-id       # optional
# GOOGLE_CLIENT_SECRET=your-google-client-secret  # optional

# 8. Start both the Worker API (port 8787) and Vite dev server (port 5173) in parallel
npm run dev
```

The `dev` script runs `concurrently` with two processes: `wrangler dev --port 8787` for the API and `vite --config web/vite.config.ts` for the frontend. Vite's proxy config routes `/api` requests to the Worker during development, so the SPA and API behave identically to production.

## Deploying to Cloudflare

Before the first production deploy, apply the migrations to the remote D1 database and set the required secrets:

```bash
# Apply migrations to production D1
npm run db:apply:prod
# (Runs: wrangler d1 migrations apply your-todo-db --remote)

# Set production secrets (Wrangler will prompt for the values)
npx wrangler secret put BETTER_AUTH_SECRET
npx wrangler secret put BETTER_AUTH_URL
# If using Google OAuth:
npx wrangler secret put GOOGLE_CLIENT_ID
npx wrangler secret put GOOGLE_CLIENT_SECRET

# Build the frontend and deploy the Worker
npm run deploy
# (Runs: vite build → wrangler deploy --minify)
```

Wrangler bundles the Worker, uploads the compiled assets to the `./public` directory binding, and prints a live URL like `https://my-todo-worker.<your-subdomain>.workers.dev`.

## Weighted Effectiveness Score

The Stats page computes a productivity score that weights tasks by priority rather than counting them equally. Each todo carries a weight: `high = 3`, `med = 2`, `low = 1`. The score for a given period is `(sum of weights of done todos) / (sum of weights of all non-archived todos) × 100`. A week where you complete three high-priority items and skip three low-priority ones scores higher than one where you do the reverse — which better reflects actual productivity:

```typescript
// src/routes/stats.ts (core calculation)
for (const r of rows) {
  if (r.status === 'archived') continue;
  const w = PRIORITY_WEIGHT[r.priority];  // { high: 3, med: 2, low: 1 }
  breakdown[r.priority].total++;
  weightedTotal += w;
  if (r.status === 'done') {
    breakdown[r.priority].done++;
    weightedDone += w;
  }
}

const score = weightedTotal === 0 ? null : (weightedDone / weightedTotal) * 100;
```

## When To Use This Architecture

This pattern works well when:

- You want a single deployment unit covering frontend, API, database, and AI with zero infrastructure management.
- The user base is globally distributed and low-latency reads matter.
- Your workload is moderate-write and read-heavy — todo apps, personal tools, small team dashboards.
- You want a built-in AI feature without adding a separate provider account or managing API keys externally.

Consider alternatives when:

- Your database write volume is high enough to hit D1's rate limits (currently 1,000 rows/s on paid plans).
- You need complex SQL features beyond SQLite's capabilities — window functions for most use cases are fine, but complex stored procedures or pg-specific extensions are not available.
- Your team needs fine-grained staging/production environment isolation beyond what Wrangler's `env` blocks provide.

## Final Thoughts

What makes this stack compelling is not any individual technology — it is that D1, KV, Workers AI, and static asset hosting are all first-class bindings inside a single Worker. You write one `wrangler.jsonc`, run one `npm run deploy`, and get a globally distributed, authenticated, AI-enhanced application with no servers to manage. The Drizzle + Zod combination keeps the schema, queries, and HTTP validation layer consistent across the codebase, which pays off quickly as the feature set grows.
